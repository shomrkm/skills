## 2. APIリファレンス（Rails controllerを想定）

```markdown
## POST /api/v1/evaluation_sheets/:id/submit

評価シートを「下書き」から「提出済み」に遷移させる。

### リクエスト

| パラメータ | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | integer | ✔ | 評価シートID |
| `comment` | string | – | 提出時コメント（任意） |

### レスポンス（成功時 200）

\`\`\`json
{
  "id": 123,
  "status": "submitted",
  "submitted_at": "2026-07-12T10:00:00+09:00"
}
\`\`\`

### レスポンス（失敗時 422）

必須項目が未入力の場合、以下を返す。

\`\`\`json
{
  "errors": [
    { "field": "rating", "message": "評価点を入力してください" }
  ]
}
\`\`\`

### 備考

- 「確定済み」ステータスのシートに対して呼ぶと409を返す。
- 提出後の差し戻しは `POST /api/v1/evaluation_sheets/:id/reject` を参照。
```

**書くときの注意**: リファレンスは物語調にしない。正常系・異常系・制約を淡々と、
検索しやすい見出しで並べる。「なぜこの設計か」は概念ドキュメント側に書き、
ここには書かない（Single Source of Truth）。

## 3. チュートリアル / How-to（React + TypeScriptを想定）

```markdown
# 評価フォームにカスタムバリデーションを追加する

このガイドでは、`useEvaluationForm`フックに新しいバリデーションルールを
追加する手順を説明する。所要時間の目安: 10分。

## 前提条件

- ローカル環境が `bin/setup` 済みであること
- `packages/evaluation-form` の既存テストが通っていること

## 手順

### 1. バリデーションルールを定義する

\`\`\`ts
// packages/evaluation-form/src/validators/ratingRange.ts
export const ratingRange = (value: number) =>
  value >= 1 && value <= 5 ? undefined : "評価点は1〜5で入力してください";
\`\`\`

### 2. フックに登録する

\`\`\`ts
const { errors } = useEvaluationForm({
  validators: { rating: ratingRange },
});
\`\`\`

### 3. 動作確認する

\`\`\`bash
pnpm test packages/evaluation-form
\`\`\`

`rating`フィールドに0や6を入力し、エラーメッセージが表示されることを確認する。

## うまくいかない場合

- エラーが表示されない → `validators`のキー名がフィールド名と一致しているか確認する。
```

**書くときの注意**: 手順は必ず実際に手を動かして検証してから書く。「〜のはず」で
書かれた手順は、読者が最初に詰まる最大の原因になる。

## 4. 概念ドキュメント / ADR（設計判断の記録）

```markdown
# 評価軸の保持方法にJSONBを採用した理由

## 背景

評価軸（評価項目・配点）はテナントごとにカスタマイズ可能で、追加・変更の頻度が高い。

## 検討した選択肢

1. **正規化テーブル（evaluation_axes等）**: 構造化クエリはしやすいが、
   軸の追加・変更のたびにマイグレーションが必要。
2. **JSONBカラム**: スキーマ変更なしで柔軟な構造を持てる。ただし集計クエリが複雑になる。

## 決定

評価軸は`evaluation_sheets.axes`のJSONBカラムに保持する。

## 理由

- 評価軸の追加・変更頻度が高く、マイグレーションのコストが運用上のボトルネックになっていた。
- 集計は月次バッチで別テーブルに書き出す方式で解決できる見込みが立った。

## トレードオフとして受け入れたこと

- JSONBの内容に対する厳密なDB制約は張れない。アプリケーション層のバリデーション
  （`EvaluationAxesValidator`）で担保する。
```

**書くときの注意**: ADRは「何を選んだか」より「他に何を検討し、なぜそれを選ばなかったか」
の方が将来の読者にとって価値が高い。
