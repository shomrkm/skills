## Ch.1 読者を理解する（Understanding your audience）

- **the curse of knowledge（知識の呪い）**: 書き手はプロダクトの内部を知り尽くしているため、
  読者にとって何が自明でないかを見落としやすい。これが読みにくいドキュメントの最大の原因。
- **ユーザー像の初期スケッチ**: 読者の役割（新人エンジニア、社外の利用者、運用担当者など）、
  技術レベル、抱えているタスクを簡単に書き出す。完璧なペルソナ資料は不要で、実在するユーザー像
  に近い解像度でよい。
- **ユーザー理解の検証**: 想定した読者像が正しいか、実際のユーザーに聞く・使ってもらう・
  既存の問い合わせ内容を見る、などで検証する。
- **ユーザー調査の結果を要約する**: 調査結果は箇条書きで簡潔にまとめ、書く際にすぐ参照できる
  形にしておく。
- **friction log（フリクションログ）**: 実際に自分（または読者役の人）が製品/手順を触りながら、
  詰まった箇所・疑問に思った箇所を時系列でそのまま記録したもの。ドキュメントで最も厚く書くべき
  箇所を客観的に特定できる。

**社内ドキュメントへの応用**: 大掛かりな調査は不要。「このドキュメントを最初に読むのは誰で、
その人は今何につまずいているか」を1〜2文で書き出すだけでも効果がある。

## Ch.2 ドキュメントを計画する（Planning your documentation）

コンテンツタイプごとに役割が異なるため、混同すると読みにくくなる。

- **コードコメント**: 「何をしているか」ではなく「なぜそうしているか」を書く。コードを読めば
  わかることは書かない。
- **README**: プロジェクトの入口。概要、セットアップ手順、最小限の使い方を最短距離で示す。
- **Getting started（はじめに）ドキュメント**: 読者が最初の成功体験（Hello World相当）に
  最短でたどり着けるようにする。
- **概念ドキュメント（Conceptual）**: 「なぜ」「何」を説明する。アーキテクチャ、設計判断の
  背景、用語の定義など。手順は書かない。
- **手順ドキュメント（Procedural）**: 特定の作業を完了させるための順序立った手順。
  チュートリアル（学習目的、ステップバイステップ）とHow-to（特定タスクの達成）を区別する。
- **リファレンスドキュメント（Reference）**: 網羅性と検索性を優先する。API仕様、設定項目、
  エラーコード一覧など。物語的な説明は避け、事実を整理して並べる。

**計画のコツ**: 書く前に、これから書くものがどのタイプかを決める。1つの文書に複数のタイプの
情報（概念説明＋手順＋リファレンス）を詰め込むと、どの読者にも中途半端になる。

<a id="ch3"></a>
## Ch.3 下書きする（Drafting documentation）

- **白紙の恐怖への対処**: 完璧な最初の文を書こうとしない。アウトライン（見出しの箇条書き）
  から始め、埋められるところから埋める。
- **執筆しやすい環境を作る**: 集中して書ける時間・場所を確保する、資料を先に手元に集めておく、
  といった準備が結果的に速く書けることにつながる。
- **アウトラインを作る**: 見出しの構造を先に決め、読者がタスク順・疑問順にたどれるかを
  文章化する前に検証する。
- **スキャン可能に書く（Writing for skimming）**: 読者の大半は上から下まで読まない。
  見出し・箇条書き・太字・短い段落を使い、拾い読みでも要点が伝わる構造にする。
- **行き詰まったときの対処**: 完璧な文章を目指さず、まずコード例や手順の箇条書きなど
  「書ける部分」から埋める。
- **テンプレートの活用**: ドキュメントタイプごとの型（README、チュートリアルなど）を使うと、
  構成に悩む時間を減らせる。
- **初稿の完成**: 初稿は「完璧」ではなく「全体の骨格が揃っている」ことをゴールにする。

<a id="ch4"></a>
## Ch.4 編集する（Editing documentation）

- **読者のニーズに合わせて編集する**: 「自分が書きたかったこと」ではなく「読者が必要として
  いること」を基準に編集する。
- **編集アプローチを分ける**: 一度に全部を直そうとしない。内容（正確性）→構成（順序・粒度）
  →文章表現（明快さ・簡潔さ）→仕上げ（誤字脱字・フォーマット）と、パスを分けて編集する。
- **編集プロセスを作る**: 誰が・いつ・何を基準にレビューするかを決めておくと、レビューの
  質と速度が安定する。
- **フィードバックの受け取り方**: フィードバックは「読者にどう伝わったか」の情報として受け取る。
  防御的にならず、指摘の背景（なぜそう感じたか）を聞く。
- **良いフィードバックの与え方**: 「わかりにくい」だけでなく、「どこで」「何を」「なぜ」
  わかりにくかったかを具体的に伝える。可能なら代案も添える。

<a id="ch5"></a>
## Ch.5 コードサンプルを組み込む（Integrating code samples）

- **コードサンプルの種類**: 最小再現例（motivating example）、実行可能なフルサンプル、
  抜粋（snippet）など、目的に応じて使い分ける。
- **良いコードサンプルの原則**: 実際に動くこと、最小限であること（余計な依存を持ち込まない）、
  現実的なユースケースを示すこと、コピペしてすぐ試せること。
- **コードサンプルの設計**: エラーハンドリングを省略しすぎない（本番で使えない誤った印象を
  与えない）一方、本質と関係ない複雑さは削る。
- **コードサンプルのツール**: サンプルを自動テストで検証する、実行可能な環境（サンドボックス等）
  を用意する、など陳腐化を防ぐ仕組みを検討する。

<a id="ch6"></a>
## Ch.6 ビジュアルコンテンツを加える（Adding visual content）

- **文章で足りないときに使う**: 関係性（データフロー、依存関係、状態遷移、アーキテクチャ）は
  図の方が伝わりやすい。逆に、図にしても伝わらない情報を無理に図にしない。
- **ビジュアルコンテンツが作りにくい理由**: 作成・保守のコストが高く、非デザイナーには
  ハードルが高い。だからこそ「本当に必要なところ」に絞る。
- **スクリーンショット**: UIの変更で陳腐化しやすいため最小限に。使う場合は何に注目すべきかを
  キャプションで明示する。
- **ダイアグラムの種類**: フローチャート、シーケンス図、アーキテクチャ図など、伝えたい関係性の
  種類に応じて図の種類を選ぶ。
- **ビジュアルコンテンツのレビュー・保守**: 文章と同様にレビュー対象にし、実装が変わったら
  更新する運用ルールを決めておく。

## Ch.7〜11 公開・フィードバック・測定・整理・保守（要約）

社内ドキュメントでは、これらの章を全部フルに適用する必要はない。実務で押さえるべき最小限:

- **Ch.7 公開**: 機能リリースとドキュメント公開のタイミングを揃える。PRテンプレートに
  「関連ドキュメントを更新したか」欄を入れると崩れにくい。
- **Ch.8 フィードバック**: 問い合わせが多い箇所 = ドキュメントの弱い箇所。サポートやSlackの
  質問チャネルを、次に厚く書く場所を決める friction log として使う。
- **Ch.9 品質測定**: 社内文書では厳密な指標より、「同じ質問が繰り返し来なくなったか」の
  定性判定で十分なことが多い。
- **Ch.10 整理**: 読者のタスク動線（README → Getting Started → 概念 → リファレンス）で
  文書間をつなぐ。書き手の頭の中の分類で並べない。
- **Ch.11 保守**: **Single Source of Truth** を守る（同じ事実を複数箇所に書かない）。
  古くなった記述は残さず削除。リンク切れ・コードサンプルの動作確認は自動化を検討。

社外向けドキュメントの本格運用（多言語化、コントリビューション受付、バージョン管理など）が
必要になったら、原著Ch.7〜11とAppendix Aを参照。専門のテクニカルライターに依頼する
判断基準は下の Appendix にまとめてある。

## Appendix: 専門家に頼むべきタイミング（原著 Appendix A より）

以下のような状況では、社内の片手間対応ではなく専門のテクニカルライター/ドキュメントエンジニアを
検討する価値がある、と原著は述べている。

- 新しい読者層（例: 非エンジニアの利用者）に対応する必要が出てきた
- サポートへの問い合わせを減らすことが経営目標になっている
- 大規模なドキュメントリリースを管理する必要がある
- 情報アーキテクチャ全体の作り直しが必要
- 多言語対応（国際化・ローカライゼーション）が必要
- ソフトウェアのバージョニングに合わせてドキュメントもバージョン管理する必要がある
- ユーザーからのドキュメント貢献（コントリビューション）を受け付ける
- ドキュメントをオープンソース化する
